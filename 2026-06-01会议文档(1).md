# 会议总览

1.  让大家先知道我们接下来要做的“职业蓝图”项目到底是什么，为什么值得做，以及大家在里面可以获得什么
    
2.  先统一一些基础工作方式和工具认知。比如产品是怎么从一个想法变成真实页面的，项目周期怎么推进，Git、Claude Code、Codex、网络代理这些工具分别解决什么问题。
    

> 以后真正重要的不是“我用过什么工具”或“我做过什么事情”，而是我能不能发现问题、拆解问题、借助工具解决问题，并且把结果交付出来。

## 一、我们要做的项目是什么

职业蓝图：面向的是学生和刚进入职场的人，解决的是一个很常见的问题：

> 我听过很多职业名字，但我其实并不知道这些岗位每天到底在做什么，也不知道自己适不适合、该怎么开始、AI 会不会影响这个岗位。

比如很多同学会听过这些岗位：

1.  产品经理
    
2.  AI 产品经理
    
3.  前端开发
    
4.  后端开发
    
5.  数据分析师
    
6.  UI 设计师
    
7.  新媒体运营
    
8.  商务 BD
    

但真的问起来，很多人其实说不清：

1.  这个岗位每天具体做什么？
    
2.  它需要什么能力？
    
3.  入门应该学什么？
    
4.  它和其他岗位怎么协作？
    
5.  AI 会替代它的哪些工作，又会增强哪些工作？
    
6.  我现在如果想了解这个岗位，下一步该做什么？
    

职业蓝图要做的，就是把这些问题讲清楚。

## 二、职业蓝图第一阶段做什么

第一阶段我们不做复杂的大平台，也不做招聘、简历、社区或者严肃测评。

我们先做一个小而清晰的 MVP。

MVP 可以理解成“最小可用版本”：不是把所有功能都做完，而是先做出一个能被真实用户体验、能验证方向的版本。

第一阶段先跑通几个岗位：

1.  产品经理
    
2.  AI 产品经理
    
3.  前端开发
    
4.  后端开发
    

每个岗位会产出几类东西：

| 内容 | 说明 |
| --- | --- |
| 岗位资料 | 来自真实 JD、资料、访谈或调研 |
| 职业卡片 | 用学生能看懂的话解释岗位 |
| 职业关系 | 说明这个岗位和其他岗位怎么连接 |
| AI 影响 | 说明 AI 会替代、增强或创造哪些机会 |
| 页面展示 | 把内容放到产品页面里，而不是停留在文档 |
| 测试反馈 | 找真实用户看能不能理解、哪里看不懂 |

我们的目标不是写一堆“看起来很完整”的文档，而是做出一个真实可看的产品雏形。

## 三、大家参与这个项目能获得什么

这个项目不是让大家来“帮忙”。

每个人都应该在项目里拿到可以沉淀的能力和作品。

### 产品方向可以获得

1.  如何把模糊想法拆成页面和功能。
    
2.  如何写需求、流程和验收标准。
    
3.  如何判断用户到底看不看得懂。
    
4.  如何用 AI 辅助做产品分析和内容初稿。
    

### 研发方向可以获得

1.  如何阅读一个项目的目录和文档。
    
2.  如何理解页面、数据和功能之间的关系。
    
3.  如何用 Claude Code、Codex 等工具辅助开发。
    
4.  如何使用 Git 做版本管理和协作。
    
5.  如何定位问题、看报错、验证修改。
    

### 内容和运营方向可以获得

1.  如何查资料并保留来源。
    
2.  如何把复杂职业信息变成普通学生能看懂的表达。
    
3.  如何把用户反馈整理成改进建议。
    

## 四、我们真正训练的能力

职业蓝图表面上是在做职业认知产品，但真正训练的是 4 种能力。

### 1. 理解问题

不是别人给你一句话，你就立刻开始做。

你要先问：

1.  这个任务要解决谁的问题？
    
2.  用户为什么需要它？
    
3.  如果做完了，怎样算有效？
    

### 2. 拆解问题

大部分任务一开始都是模糊的。

比如“做一个职业详情页”听起来很大，但可以拆成：

1.  先找真实资料。
    
2.  再总结岗位任务。
    
3.  再写成 8 张职业卡片。
    
4.  再整理成数据结构。
    
5.  最后放到页面里测试。
    

### 3. 使用工具

AI、Git、Claude Code、Codex、搜索、网络代理，这些都是工具。

工具本身不是目的。工具的作用是让我们更快理解问题、更快产出初稿、更快检查错误。

### 4. 交付结果

最后一定要有结果。

结果可以是：

1.  一份文档。
    
2.  一张表格。
    
3.  一段代码。
    
4.  一个页面截图。
    
5.  一份测试记录。
    
6.  一份用户反馈总结。
    

只说“我研究了”“我看了”“我试了一下”，不算完整交付。

## 五、产品和项目周期

一个产品通常不是一上来就写代码。

它大概会经历这些阶段：

| 阶段 | 要解决的问题 | 产出 |
| --- | --- | --- |
| 发现问题 | 用户到底卡在哪里 | 用户问题、访谈记录、场景描述 |
| 定义需求 | 我们先解决哪一个问题 | 需求说明、页面范围、验收标准 |
| 设计方案 | 用户怎么用，页面怎么走 | 流程、原型、文案、内容结构 |
| 技术实现 | 怎么把方案变成能运行的东西 | 页面、数据、接口、部署 |
| 测试验证 | 用户能不能用，哪里看不懂 | 问题清单、修复记录、测试报告 |
| 复盘迭代 | 哪些有效，下一步做什么 | 复盘文档、下一轮任务 |

职业蓝图现在处在第一个阶段：启动、训练和小样验证。

## 六、今天要认识的常用工具

### 1. AI 对话工具

比如 ChatGPT、Gemini等。

它们适合做：

1.  解释概念。
    
2.  拆解任务。
    
3.  生成初稿。
    
4.  改写表达。
    
5.  总结资料。
    

但要注意，AI 不是最终答案。

AI 容易出现的问题包括：

1.  编造事实。
    
2.  表达很顺，但内容很空。
    
3.  没有来源。
    
4.  看起来完整，但不符合真实业务。
    

所以我们使用 AI 的方式应该是：

> 让 AI 帮我们起步，但最后由人来判断、修改和负责。

### 2. Claude Code / Codex

Claude Code 和 Codex 这类工具，不只是聊天工具，它们更适合在项目里工作。

它们可以帮你：

1.  阅读项目目录。
    
2.  理解文档和代码。
    
3.  找某个功能在哪里。
    
4.  修改文档或代码。
    
5.  根据报错分析问题。
    
6.  帮你规划下一步怎么做。
    

但使用这类工具时，不要只说“帮我做一下”。

你要尽量说清楚：

1.  项目背景是什么。
    
2.  你要解决什么问题。
    
3.  你希望它看哪些文件。
    
4.  你不希望它改什么。
    
5.  你希望输出成什么格式。
    
6.  怎么判断结果是合格的。
    

### 3. Git

Git 是团队协作中非常重要的版本管理工具。

你可以把它理解成：

> Git 帮我们记录项目每一次重要修改，让大家可以一起协作，也能回头看是谁在什么时候改了什么。

先理解 4 个概念：

| 概念 | 说明 |
| --- | --- |
| 仓库 | 项目的完整文件夹和历史记录 |
| 分支 | 一条独立的修改线 |
| 提交 | 一次可追溯的修改记录 |
| 合并 | 把某条分支的修改合回主线 |

刚开始不用掌握所有命令，但至少要知道 Git 解决的是协作问题。

没有 Git，几个人同时改项目，很容易出现：

1.  不知道谁改了什么。
    
2.  新内容覆盖旧内容。
    
3.  出错后不知道怎么回退。
    
4.  多个人的版本不一致。
    

### 4. 网络代理

网络代理不是能力本身，但它会影响工具能不能正常使用。

有些 AI 工具、开发平台、文档网站和代码服务需要稳定网络。如果网络或代理有问题，可能会出现：

1.  网页打不开。
    
2.  登录失败。
    
3.  工具连接超时。
    
4.  下载依赖失败。
    
5.  AI 工具无法响应。
    

遇到这类问题时，不要马上判断“代码坏了”或“工具坏了”。

要先分清楚：

1.  是网络问题？
    
2.  是账号问题？
    
3.  是代理配置问题？
    
4.  是工具本身的问题？
    
5.  还是代码真的有问题？
    

### 5. 搜索和资料来源

职业蓝图里的职业内容不能只靠想象，也不能只靠 AI 编。

资料来源优先级大概是：

1.  真实招聘 JD。
    
2.  公司官网或岗位介绍。
    
3.  行业报告。
    
4.  真实访谈。
    
5.  AI 总结。
    

AI 总结只能做辅助，不能当作唯一来源。

如果你写“产品经理需要沟通能力”，这句话不难。

但更好的交付是：

1.  你看了哪些 JD。
    
2.  这些 JD 里反复出现了哪些任务。
    
3.  哪些能力是高频出现的。
    
4.  你把它们翻译成了学生能看懂的话。
    

## 七、我们接下来怎么工作

### 1. 接任务时先问 5 个问题

每次拿到任务，不要立刻开干，先问清楚：

1.  这个任务要解决什么问题？
    
2.  最终交付物是什么？
    
3.  截止时间是什么？
    
4.  验收标准是什么？
    
5.  我需要谁配合？
    

### 2. 做任务时先拆 3 步

比如任务是“写产品经理职业卡片”。

可以先拆成：

```text
第一步：找 5-10 条真实 JD 和资料。
第二步：用 AI 总结常见任务、技能和工具。
第三步：人工改写成学生能看懂的 8 张卡片。

```

### 3. 交付时带 4 样东西

每次交付尽量带上：

1.  最终交付物。
    
2.  使用了什么工具。
    
3.  AI 或资料来源是什么。
    
4.  自己检查过哪些问题。
